## Transport and capability requirements applied

- Stdio uses newline-delimited UTF-8 JSON-RPC, protocol-only stdout, and stderr logging.
- Streamable HTTP uses one `/mcp` POST/GET endpoint plus DELETE for session termination. It is stateful to support progress, cancellation, server notifications, SSE resumption, and secure random sessions.
- Sessions are bound to authenticated principals and expire; they are never authentication credentials.
- Tools use strict schemas and explicit read-only/destructive/idempotent/open-world annotations. Hints do not replace enforcement and do not require a client to show a confirmation prompt for each call.
- Long work observes `AbortSignal` and sends monotonic progress only when the client supplies a progress token.
- Large results are content-addressed resources. Binary reads are bounded/chunkable because resource links defer retrieval but are not a byte-stream transport.
- The server does not register MCP prompts; coding agents choose when to call the ten tools from their own task context.

### Write-policy conclusion

MCP separates server-advertised tool metadata from client-host policy. A server can truthfully mark a source-changing tool with `readOnlyHint: false` and `destructiveHint: true`, but it cannot require, suppress, or bypass a host confirmation dialog. The host decides how to present or constrain the call; the server independently enforces its configured workspace, authentication, scope, validation, journaling, and recovery boundaries.

For this server, every configured mod exposes one-call `hoi4.focus_rewrite`, `hoi4.gui_rewrite`, and `hoi4.map_rewrite` tools with destructive annotations. The tools internally plan, validate, journal, apply, and post-validate, restoring original bytes on failure. Transaction, approval, and rollback tools are not registered.

## Remote security requirements applied

- Loopback is the default bind address.
- Present origins and Host are validated; request body, concurrency, rate, session, and expiry limits are enforced.
- Every HTTP method authenticates. Static environment bearer tokens are loopback-only.
- Non-loopback operation requires HTTPS and OAuth/OIDC resource-server JWT verification: issuer, JWKS, audience, algorithms, expiry, subject, and required scopes.
- Protected Resource Metadata is exposed when OAuth is configured.
- Principal-to-workspace grants are server-side; token passthrough, query-string tokens, cross-principal sessions, and cross-workspace transactions are rejected.

## Registry requirements applied

- `package.json.mcpName` exactly matches `server.json.name`.
- The public npm package version, server version, and Registry version are exact and synchronized.
- Registry versions are immutable; release metadata is validated before publishing.
- npm is published first because the Registry stores discovery metadata rather than package bytes.
- The release workflow uses GitHub OIDC for `mcp-publisher` and npm provenance.
- npm trusted publishers created after May 20, 2026 require an explicit allowed action; this package selects `npm publish`, uses a GitHub-hosted runner with `id-token: write`, and does not restore a package-manager cache in release jobs.
- The pinned official publisher's live `validate` operation accepts this release's `server.json` before publication.
